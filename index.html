1. Swift Language Fundamentals

1.1 What is the difference between let and var?

let binds a constant; var binds a mutable variable. The architectural point is that for value types let freezes the entire value (you cannot mutate a let struct's properties), whereas for reference types let only freezes the pointer, not the object's internal state. Default to let: it documents intent, lets the optimizer assume no mutation, and removes a whole class of accidental-write bugs. Reaching for var should be a deliberate signal that something genuinely changes over time.

class Person {
    var name: String
    init(name: String) {
        self.name = name
    }
}
// This is only fezzeing the pointer, not the internal state of object which is name.
let person: Person = Person(name: "Jack")
print(person.name)
person.name = "Rose"
print(person.name)

//Output:
Jack
Rose

Refs: The Swift Programming Language: The Basics

1.2 What are optionals and why do they exist?

An optional T? is sugar for the Optional<T> enum (.some(T) or .none). It encodes "value may be absent" directly in the type system, so the compiler forces you to handle absence. Fundamentally this converts the billion-dollar null-pointer problem from a runtime crash into a compile-time obligation, which is one of Swift's biggest safety wins over Objective-C's silent nil messaging.


var optional: String?
print(optional)
print(optional?.isEmpty)
optional = "Hello"
print(optional)
print(optional?.isEmpty)
print(optional!)
if let optional = optional {
    print(optional)
}

// Output:
nil
nil
Optional("Hello")
Optional(false)
Hello
Hello

``

1.3 What are the ways to unwrap an optional?

Optional binding (if let, guard let), nil-coalescing (??), optional chaining (a?.b?.c), pattern matching (if case .some(let x)), and forced unwrap (!, only when non-nil is provable). Prefer guard let for preconditions because the unwrapped value stays in scope and the failure path is explicit and early.

guard let user = currentUser else { return }   // user is non-optional below
let name = user.nickname ?? "Guest"

1.4 What is the difference between ?? and optional chaining?

Optional chaining short-circuits a chain of accesses and yields an optional; ?? supplies a fallback when an optional is nil. They compose cleanly, which is the idiomatic way to express "drill in, but default if anything along the way is missing": user?.profile?.name ?? "Guest".

1.5 What is the difference between a struct and a class?

Structs are value types (copied on assignment/passing, no inheritance, no identity, typically stack-friendly). Classes are reference types (shared by reference, support inheritance and deinit, have identity via ===, heap-allocated). The senior take: prefer structs for models because value semantics eliminate shared-mutable-state bugs and make concurrency far safer; choose a class only when you genuinely need identity, inheritance, shared mutable state, or Objective-C interop.

Refs: Swift Book: Structures and Classes

1.6 When do you choose a value type over a reference type?

Choose value types when instances are defined by their data and are interchangeable (coordinates, money, DTOs), when you want predictable copies and thread-safety from immutability, and when you do not need inheritance. Choose reference types for shared, identity-bearing, or long-lived mutable objects (a controller, a cache, a network client). A good heuristic: "is this a thing, or a value of a thing?"

1.7 What is copy-on-write (COW) and how does it work?

Swift's Array, Dictionary, Set, and String are value types but share one backing buffer until a mutation occurs; on mutation, if the buffer is uniquely referenced it is changed in place, otherwise it is copied first. This gives value semantics at reference-sharing cost. You can implement COW for custom types with a private class buffer and isKnownUniquelyReferenced(_:), which is exactly how you build large value types without paying for copies on every pass.

1.8 Are tuples value or reference types? What are the reference types in Swift?

Tuples, structs, and enums are value types. The reference types are classes, closures, and actors. Knowing this matters because only reference types participate in ARC and can form retain cycles.

1.9 What is the difference between static and class members?

Both are type-level. static members cannot be overridden; class methods and computed properties (on classes) can be overridden by subclasses. Use class only when you want polymorphism at the type level; otherwise static is clearer and enables faster dispatch.

    }
    class func newBar() {
        print("Super: new bar")
    }

1.10 What is mutating, and why is it needed on structs?

A method that changes a value type's stored state must be marked mutating because it conceptually rebinds self to a new value. You cannot call it on a let value. Protocols that value types must satisfy with state changes declare mutating func. This is the language making value semantics explicit rather than hiding mutation.

public struct User: Codable {
    let id: Int
    let name: String
// email has to be var as it will be mutated with another funtion.

    private(set) var email: String

    // since it is modifying the value types (struct:User)'s state. need to mentioned mutating method.

    mutating func updateEmail(newEmail:String) {

        self.email = newEmail

    }

}

  

// Can't define let user as it has mutable value inside

var user =  User(id: 1, name: "Aman", email: "aman@gmail.com")

print("Before Changing email:\(user.email)")

  

user.updateEmail(newEmail: "new@gmail.com")

print("After Changing email:\(user.email)")

1.11 What are KeyPaths?

A KeyPath is a type-safe, reusable reference to a property (\User.name), with WritableKeyPath and ReferenceWritableKeyPath variants. They power users.map(\.name), SwiftUI bindings, Combine's assign(to:on:), and generic, data-driven code where you parameterize over "which property" rather than hardcoding access.

1.12 What is the difference between Any and AnyObject?

Any holds an instance of any type, including value types and functions; AnyObject holds only class instances. You constrain a protocol to classes with AnyObject (protocol Delegate: AnyObject) so it can be held weak. Both are the modern, narrower replacements for Objective-C's untyped id.

1.13 What is the difference between Equatable, Hashable, Comparable, and Identifiable?

Equatable defines ==; Hashable (refines Equatable) adds hash(into:) so values can be set members or dictionary keys; Comparable defines ordering for sorting; Identifiable provides a stable id that SwiftUI and diffing use to track a thing across updates. The compiler synthesizes Equatable/Hashable when all stored members already conform, so conformance is usually free.

#### What is a Self requirement and why is Equatable affected? Equatable's == takes two values of Self. That binds the protocol to the concrete type, making it non-usable as a bare existential historically (you couldn't compare two arbitrary Equatable boxes meaningfully)

1.14 What is the difference between == and ===?

== is value equality through Equatable. === is identity, asking whether two references point to the same instance, and applies only to classes. Conflating them is a classic bug: two distinct objects can be equal by value yet not identical.

2. Protocols, Generics & Type System

2.1 What is protocol-oriented programming?

A design approach that builds behavior from small protocols plus protocol-extension default implementations, rather than deep class inheritance. Protocols are capability contracts; extensions supply shared behavior; associated types and conditional conformance make them generic. The payoff is that value types can share behavior without a base class, avoiding fragile inheritance trees and enabling composition.

POP is a paradigm, formally introduced by Apple at WWDC 2015 ("Protocol-Oriented Programming in Swift"), where protocols are the primary unit of abstraction instead of base classes. You define _capabilities_ as protocols, supply shared behavior through protocol extensions, and compose behavior by conforming a type to multiple protocols rather than inheriting from a single superclass. It favors value types (structs/enums) over reference types and composition over inheritance.

Refs: Swift Book: Protocols, WWDC: Protocol-Oriented Programming in Swift

What problems does POP solve that classic OOP struggles with?

  • The diamond problem / rigid single inheritance — a class can inherit from only one superclass; a type can conform to many protocols.
  • Forced reference semantics — base classes drag in shared mutable state and identity. POP lets you keep value semantics (predictable copies, thread-safety, no unexpected aliasing).
  • Retroactive modeling — you can make existing types (even Int or String) conform to your protocols via extensions, without touching their source.
  • Bloated base classes — instead of one God-class with everything, you slice behavior into small focused protocols (Interface Segregation).

Name some protocols from the Swift standard library and Apple frameworks that demonstrate POP.

Standard library:

  • Equatable, Hashable, Comparable, Codable, Sequence, Collection, Identifiable, CustomStringConvertible, Error, IteratorProtocol.

SwiftUI / Combine:

  • View, ObservableObject, Publisher, Identifiable, Animatable

Which types can conform to a protocol?

  • Classes, structs, enums, and even other protocols (protocol inheritance). Actors too (Swift concurrency).
  • Only classes can additionally use a protocol's class-only features (e.g. AnyObject constraint, weak references to the protocol type).

POP vs OOP

How does POP relate to SOLID?

  • Interface Segregation → many small focused protocols instead of fat interfaces.
  • Dependency Inversion → depend on protocol abstractions, not concrete types (great for testing/mocking).
  • Open/Closed → extend behavior via protocol extensions without modifying existing types.
  • Single Responsibility → each protocol models one capability.

Composition over inheritance: explain with an example.

In OOP you might write class Bird: Animal and then get stuck when a Penguin "is a Bird" but can't fly. In POP you define orthogonal capabilities:

protocol Animal { var name: String { get } }
protocol Flyable { func fly() }
protocol Swimmable { func swim() }

struct Sparrow: Animal, Flyable { let name = "Sparrow"; func fly() {} }
struct Penguin: Animal, Swimmable { let name = "Penguin"; func swim() {} }

No awkward Penguin overriding fly() to throw an error. You only adopt the capabilities you actually have.

When would you still reach for a class / inheritance?

  • You genuinely need reference semantics / shared identity (e.g. a single source of truth, delegates held weakly).
  • Objective-C interop (UIKit, NSObject subclasses, KVO, selectors).
  • Inheritance from a framework base class you don't control (UIViewController, Operation).
  • A natural is-a hierarchy with shared stored state where subclassing is genuinely the cleanest model.

POP and OOP are not mutually exclusive — idiomatic Swift mixes both.

Can you constrain a protocol extension?

Yes — where clauses. This is extremely powerful:

extension Collection where Element: Numeric {
    func total() -> Element { reduce(.zero, +) }
}

Only collections of Numeric elements get total(). The standard library uses this everywhere.

where clauses on generics: give an example.

func allEqual<C: Collection>(_ c: C) -> Bool where C.Element: Equatable {
    guard let first = c.first else { return true }
    return c.allSatisfy { $0 == first }
}

#### What is protocol composition? Combining multiple protocols into one requirement with &:

func handle(_ x: any Codable & Identifiable) { ... }
typealias Persistable = Codable & Hashable

#### Existentials: any, some, and Type Erasure

#### What is an existential type? An existential is a value accessed _through a protocol_ rather than its concrete type — "some type, but I only know it conforms to P."

  • It's a boxed representation: the compiler creates an invisible box that holds the concrete value plus a pointer to its witness table.
  • The concrete type is erased at compile time and known only at runtime.
var pet: any Animal = Cat()   // existential box
pet = Dog()  // can hold a different concrete type later

#### some vs any — the key interview distinction.

![[Screenshot 2026-06-29 at 7.38.31 PM.png]] Rule of thumb (SwiftLee):

  • start with concrete types → move to some when you need flexibility → use any only when you must store arbitrary/heterogeneous values.

#### Q4.1 — What dispatch mechanisms does Swift have?

1. Static (direct) dispatch —

  • resolved at compile time. Fastest, inlinable.
  • Used for value types, (struct, enum, tuples)
  • final classes/methods,
  • private methods,
  • global functions, and 
  • methods declared only in a protocol/class extension.

2. Dynamic dispatch via vtable (witness table for protocols) —

  • resolved at runtime.
  • Used for overridable class methods and 
  • protocol requirements.
  • Swift uses a protocol witness table (PWT) that maps each requirement to the conforming type's implementation.

3. Dynamic dispatch via message send (objc_msgSend) — for @objc dynamic members and Objective-C interop. Most flexible, slowest.

The classic gotcha. What does this print and why?

protocol Speaker {
    func sayHello()          // requirement
}
extension Speaker {
 // default impl
    func sayHello() { print("Protocol default") }  
 // NOT a requirement
    func sayBye()   { print("Protocol bye") }        
}

struct Person: Speaker {
    func sayHello() { print("Person hello") }
    func sayBye()   { print("Person bye") }
}

let p: Speaker = Person()
p.sayHello()   // ?
p.sayBye()     // ?

//let p1: Person = Person()
let p1 = Person()
p1.sayBye()     // ?

Answer:

  • p.sayHello() → "Person hello". 
  • sayHello() _is_ a protocol requirement, so it's dispatched through the witness table to Person's implementation, even though p is typed as Speaker.
  • p.sayBye() → "Protocol bye". 
  • sayBye() is not declared in the protocol body — it only exists in the extension.
  • Calls through the protocol type are statically dispatched to the extension version. 
  • Person's sayBye() does not override it; it merely _shadows_ it when called on a Person-typed variable.

If you changed let p: Speaker to let p = Person(), then p.sayBye() would print "Person bye".

Rule of thumb: If you want a method to be a customization point, declare it in the protocol body. Otherwise extension-only methods get fixed at compile time based on the _static_ type.

###### How do you force/avoid dynamic dispatch?

  • final on a class/method forces static dispatch (perf win).
  • @objc dynamic forces message dispatch (needed for KVO, method swizzling).
  • Generic constraints (some/<T: Protocol>) often let the compiler specialize and use static dispatch, which is why generics outperform existentials.

#### Rapid-Fire Cheat Sheet

  • POP mantra: start with a protocol, prefer value types, compose don't inherit.
  • Protocol extension = default implementations + retroactive behavior.
  • Requirement (in body) = dynamic dispatch / customization point. Extension-only method = static dispatch, not overridable through the protocol type.
  • PAT = protocol with associatedtype or Self requirement.
  • some P = one hidden concrete type, fast, specialized.
  • any P = boxed existential, heterogeneous, slower.
  • any keyword = explicit existential, makes boxing cost visible (SE-0335, Swift 5.6/5.7).
  • Primary associated types (Protocol<T>, SE-0346) = constrain associated types with angle brackets at the use site.
  • Type erasure = hide associated/concrete type behind a uniform box (AnySequence, or automatic via any).
  • Witness table (PWT) = runtime map from protocol requirement → conformer's implementation.
  • Still use classes for: reference semantics, ObjC interop, framework base classes, weak delegates.

2.2 What are associated types in protocols?

An associatedtype is a placeholder the conforming type resolves (Sequence has associatedtype Element). They make protocols generic over types they do not yet know. A protocol with associated types or Self requirements historically could not be a plain existential and required generic constraints; Swift 5.7+ allows any existentials in many such cases, but the underlying constraint still shapes API design.

An associatedtype is a placeholder type inside a protocol that the conforming type fills in. Think "generics without angle brackets."

protocol Container {
    associatedtype Item
    mutating func append(_ item: Item)
    var count: Int { get }
    subscript(i: Int) -> Item { get }
}

A protocol with an associated type (or a Self requirement) is called a PAT — Protocol with Associated Type.

2.3 What is the difference between some and any?

some P is an opaque type: one concrete, compiler-known conforming type, hidden from the caller, preserving type identity with static dispatch and no boxing (this is why a SwiftUI body returns some View). any P is an existential box that can hold any conforming type at runtime, with dynamic dispatch and possible heap allocation. Use some for a single underlying type and hot paths; use any for heterogeneous storage like [any Shape]. The fundamental trade is type precision and performance versus runtime flexibility.

Refs: Swift Book: Opaque and Boxed Types

2.4 What is type erasure and when do you need it?

Type erasure hides a concrete generic type behind a uniform interface so values with different associated types can be stored together. AnyView, AnyPublisher, and AnyHashable are canonical examples. You reach for it to stabilize an API boundary, store heterogeneous conformers, or stop generic types from leaking into public signatures. Since any existentials matured, hand-written erasers are needed less, but the technique still matters for clean module interfaces.

2.5 What are generic constraints and conditional conformance?

Constraints restrict the types a generic accepts via protocol bounds and where clauses (func max<T: Comparable>). Conditional conformance (extension Array: Equatable where Element: Equatable) and constrained extensions (extension Collection where Element == Int) add behavior only when constraints hold. This is how Swift keeps generics both safe and expressive without runtime type checks.

2.6 What are protocol extensions and default implementations?

An extension on a protocol provides method/property implementations every conformer inherits and may override. This is Swift's mechanism for sharing behavior across unrelated value and reference types without a base class. A subtle gotcha: dispatch of a method only declared in the extension (not in the protocol requirements) is static, so it does not dispatch dynamically through the existential.

2.7 What are extensions and what can't you add in one?

Extensions add methods, computed properties, initializers, subscripts, nested types, and protocol conformance to any type, even one you do not own (the modern replacement for Objective-C categories). You cannot add stored properties or designated initializers to classes, and you cannot override existing members. They are the primary tool for organizing code by capability and retrofitting conformances.

2.8 What do you do when a type cannot satisfy a protocol requirement?

Provide a default in a protocol extension so conformers need not implement it, redesign the protocol into smaller pieces (interface segregation), or mark a requirement @objc optional (class-and-Objective-C-only). The architect's instinct is usually to split a fat protocol rather than to make members optional, because optional requirements push nil-checks onto every caller.

A non-escaping closure (the default) is guaranteed to run before the function returns, so it cannot be stored and allows optimizations and implicit self. An @escaping closure may outlive the call (stored as a property, dispatched async), so it must capture self explicitly and is a candidate for retain cycles. Comparison of Closure Types 1. Lifetime

  • Non-Escaping: The closure's life is strictly limited to the execution scope of the function. It is executed and discarded before the function returns.
  • Escaping (@escaping): The closure is designed to live longer than the function that received it. It is often stored in a variable, a class property, or executed inside an asynchronous task (like a network call) that completes after the function returns.

2. Requirement

  • Non-Escaping: This is the default in Swift. No extra syntax is required.
  • Escaping (@escaping): You must explicitly mark the parameter with the @escaping attribute to tell the compiler to keep the closure in memory.

3. Handling self

  • Non-Escaping: Safe to use directly. Since the function finishes execution immediately, there is no risk of a retain cycle.
  • Escaping (@escaping): Usually requires a capture list—specifically [weak self] or [unowned self]—to prevent strong reference cycles and memory leaks.

4. Performance

  • Non-Escaping: Faster. Because the compiler knows exactly when the closure is used, it can perform aggressive optimizations, such as inlining the code.
  • Escaping (@escaping): Slightly more overhead. The runtime must manage the closure's memory lifecycle, as it needs to persist even after the initial scope has exited.

3.2 What are @autoclosure and trailing closures?

@autoclosure wraps an argument expression in a closure automatically so it is evaluated lazily; it backs assert, &&/||, and ??, where you do not want to evaluate the right side unless needed. A trailing closure moves the final closure argument outside the parentheses for readability, and multiple trailing closures (Swift 5.3+) label the rest. Both are ergonomics that, used carefully, make DSL-like call sites readable.

//Trailing Closures
// From Clutter 
performAction(value: 10, completion: { print("Done") })

// To this Clean
performAction(value: 10) { 
    print("Done") 
}

//Multiple Trailing Closures **(Swift 5.3+):**
// Reads like a custom DSL
animate(duration: 0.3) {
    self.view.alpha = 0
} completion: {
    self.view.removeFromSuperview()
}

An @autoclosure automatically creates a closure from an expression. This allows you to delay (or completely skip) the evaluation of that expression without forcing the caller to write curly braces { } every time they use your function.


//Without @autoclosure:
func logIfTrue(_ condition: () -> Bool) {
    if condition() { print("Logged!") }
}

// Caller must use braces every time
logIfTrue({ 1 == 1 })


//With @autoclosure:

    if condition() { print("Logged!") }
}

// Caller writes a standard expression
logIfTrue(1 == 1)

3.3 What is the difference between a function and a method?

A function is a standalone callable; a method is a function bound to a type that receives an implicit self (instance or type method). Both are first-class values, so you can pass a method as a function reference. This first-class nature is what lets you write array.map(transform) or store callbacks without ceremony.

func doSomething() {
    print("func: Do something")
}

class MyClass {
    func doSomething() {
        print("Method: Do something \(type(of: self))")
    }
}

doSomething()
MyClass().doSomething()

//Output
func: Do something
Method: Do something MyClass

3.4 What are the core higher-order collection functions?

map transforms each element, compactMap transforms and drops nils, flatMap flattens nested sequences, filter keeps matches, and reduce folds to a single value. They replace manual loops with declarative, intention-revealing pipelines and compose well; lazy avoids building intermediate arrays in long chains. This is the functional core that makes Swift data transformations concise and less error-prone.

let activeNames = users.lazy.filter(\.isActive).map(\.name)
let total = prices.reduce(0, +)

3.5 What are associated values and raw values in enums?

Raw values give each case a backing literal of one type (enum Suit: String) with a rawValue and a failable init. Associated values attach per-case payloads (case success(Data), case failure(Error)), which is what makes Swift enums excellent state machines and result types. A case can have one or the other, not both. Modeling state as an enum with associated values makes illegal states unrepresentable, a key architectural lever.

3.6 What is an indirect enum?

indirect lets a case reference the enum type itself by boxing the recursive payload on the heap, enabling recursive structures like trees and expression grammars. Without it, a value-type enum could not have infinite size.

indirect enum Expr { case value(Int); case add(Expr, Expr) }

3.7 How does Swift error handling work?

Failable functions are marked throws and throw values conforming to Error; callers use do/catch, try? (to optional), or try! (trap). Errors propagate up the stack until caught, and defer runs cleanup on any scope exit. Swift 6 adds typed throws (throws(MyError)) for precise error contracts. The design keeps the happy path readable while making failure explicit and recoverable rather than via exceptions you can ignore.

Refs: Swift Book: Error Handling

3.8 What is Result and when is it useful?

Result<Success, Failure: Error> is an enum with .success/.failure. It captures an outcome as a value, which is ideal for callback APIs that cannot use throws, for passing errors across closure boundaries, and for storing an outcome to handle later. Bridge it with get() (which throws) or into async with a continuation. In modern code async/await reduces the need for it, but it remains useful at boundaries.

3.9 What is guard and why prefer it for preconditions?

guard requires a condition (or optional binding) to hold or it must exit the scope, keeping unwrapped values in scope and the happy path unindented. It pushes failure handling to the top of a function and reduces the "pyramid of doom," which is why senior code reads as a flat list of preconditions followed by the real work.

4. Advanced Swift Features

4.1 What are property wrappers?

A property wrapper is a generic type that intercepts a property's get/set to add reusable behavior, declared with @propertyWrapper and a wrappedValue. @State, @Published, and @AppStorage are wrappers; the projectedValue (the $ form) exposes a secondary interface like a Binding or Publisher. Architecturally they let you factor cross-cutting storage concerns (persistence, validation, observation) out of every property into one reusable type.

@propertyWrapper struct Clamped {
    private var value: Int; let range: ClosedRange<Int>
    init(wrappedValue: Int, _ range: ClosedRange<Int>) { self.range = range; value = min(max(wrappedValue, range.lowerBound), range.upperBound) }
    var wrappedValue: Int { get { value } set { value = min(max(newValue, range.lowerBound), range.upperBound) } }
}

4.2 What are result builders?

A @resultBuilder transforms a sequence of statements in a closure into one aggregated value via static methods (buildBlock, buildOptional, buildEither, buildArray). SwiftUI's ViewBuilder uses this to give the declarative DSL where you just list views. The fundamental idea is compile-time transformation of ordinary-looking code into a data structure, which is what makes embedded DSLs feel native.

4.3 What are @dynamicMemberLookup and @dynamicCallable?

@dynamicMemberLookup resolves arbitrary member access at runtime through subscript(dynamicMember:) (useful for JSON wrappers or interop with dynamic languages); @dynamicCallable lets an instance be called like a function. Both trade compile-time safety for flexibility and should be used sparingly, only where the dynamism genuinely buys clarity.

4.4 What are Swift macros?

Macros (Swift 5.9+) generate code at compile time from annotations, expanding into ordinary, type-checked Swift you can inspect. Freestanding macros (#Preview, #expect) and attached macros (@Observable, @Model) eliminate boilerplate while staying safe and debuggable, unlike runtime reflection. They are how modern frameworks (SwiftData, Observation, Swift Testing) deliver concise APIs without sacrificing type safety.

Refs: Swift Book: Macros, WWDC23: Write Swift macros

5. Memory Management & ARC

5.1 How does ARC work?

Automatic Reference Counting tracks the strong reference count of each class instance; the compiler inserts retain/release at compile time, and when the count hits zero the instance is deallocated and deinit runs. It is deterministic (no stop-the-world collector) and cheap, which suits mobile, but it cannot detect cycles, so breaking them is the programmer's job. Only reference types (classes, closures, actors) are reference-counted.

Refs: Swift Book: Automatic Reference Counting

5.2 What is a retain cycle and how do you break it?

A cycle exists when two or more objects hold each other strongly, so no count reaches zero and all leak. Classic cases: a closure stored on self that captures self, a delegate held strongly, parent/child both strong. Break it with weak/unowned references and [weak self]/[unowned self] capture lists. The mental model: every strong reference is an ownership claim, and ownership must form a tree, not a loop.

5.3 What is the difference between weak and unowned?

weak is optional and auto-nils when the referent deallocates, so it is safe but requires optional handling; use it when the referent can legitimately disappear (delegates, async callbacks, parent links). unowned is non-optional and assumes the referent outlives the reference; accessing it after deallocation crashes, so use it only when lifetimes are guaranteed (a closure that cannot exist after self). Choosing wrong with unowned trades a leak for a crash, so default to weak unless you can prove the lifetime.

client.fetch { [weak self] result in
    guard let self else { return }
    self.handle(result)
}

5.4 When does a closure capture self strongly, and why is it a problem?

Closures capture referenced variables strongly by default. If self owns the closure (a stored completion handler, a Combine sink, a Timer target) and the closure captures self, you get a cycle. Use [weak self]. One nuance: a one-shot DispatchQueue.main.async that does not get stored on self does not create a permanent cycle, but [weak self] is still common defensively for objects that may be torn down before the closure runs.

5.5 What is autoreleasepool and when do you need it?

An autorelease pool collects autoreleased objects and drains them at its end; the main run loop drains one per iteration automatically. In tight loops creating many temporary bridged Objective-C objects (image or Data processing), wrapping the body in autoreleasepool { } bounds peak memory by freeing each iteration instead of at loop end.

5.6 What is deinit and when does it run?

deinit is a class's deinitializer, run automatically just before deallocation when the last strong reference goes away. Use it to release manual resources (observers, timers, file handles) and, during development, to log and confirm objects actually deallocate (a quick leak detector). Value types have no deinit.

5.7 What is the difference between shallow and deep copy?

A shallow copy duplicates references, so both copies share the referenced objects; a deep copy duplicates the referenced objects too, making them independent. Swift value types give deep, independent copies automatically via COW; for graphs of classes you implement copying yourself (or conform to NSCopying). This is why modeling shared data as value types removes an entire category of aliasing bugs.

5.8 What does EXC_BAD_ACCESS mean and what causes it?

It means code touched memory it does not own, usually a dangling pointer. In Swift the common causes are a dangling unowned reference used after its target deallocated, misuse of Unsafe/Unmanaged pointers, or over-release in bridged Objective-C/C. Address Sanitizer and Zombie Objects help localize it.

5.9 When does ARC NOT protect you?

ARC cannot break reference cycles (you must annotate weak/unowned), cannot save you from a dangling unowned access, and does not manage raw memory from Unmanaged/unsafe pointer APIs or C interop. It also will not stop a long-lived closure from extending an object's lifetime. ARC automates counting, not ownership design.

5.10 How do you respond to a memory warning, and detect leaks?

On a warning, free caches and regenerable data (UIKit delivers didReceiveMemoryWarning; SwiftUI mostly relies on the system). To find leaks, use the Memory Graph Debugger to spot retain cycles, the Leaks and Allocations instruments to track growth, and deinit logging to verify deallocation. Reproduce by repeatedly pushing/popping a screen while watching the live object count.

Refs: WWDC21: Detect and diagnose memory issues

5.11 What happens if you capture self strongly in a long-lived Task or Combine subscription?

The task or subscription keeps self alive until it finishes or is cancelled, which can leak a view model or controller well past its useful life. Use [weak self] in long-lived closures, store Task/AnyCancellable handles, and cancel them in deinit or onDisappear. Lifetime of async work is now part of memory management, not separate from it.

6. Concurrency & Multithreading

6.1 What is the difference between concurrency and parallelism?

Concurrency is structuring a program so multiple tasks can make progress in overlapping periods (interleaving); parallelism is literally running tasks at the same instant on multiple cores. Concurrency is a design property; parallelism is an execution property. A well-structured concurrent program runs in parallel when hardware allows, but correctness must not depend on it.

6.2 Why must UI updates happen on the main thread?

UIKit and SwiftUI are not thread-safe and assume single-threaded access on the main thread, which also drives the run loop and rendering. Touching UI off the main thread causes glitches, crashes, or undefined behavior. Modern Swift expresses this with the @MainActor, so the compiler can enforce it rather than relying on discipline.

6.3 What is the difference between a race condition and a data race?

A data race is concurrent access to the same memory with at least one write and no synchronization, which is undefined behavior. A race condition is a broader correctness bug where the result depends on timing or ordering even without memory unsafety. Swift's actor isolation and the Swift 6 data-race-safe model aim to eliminate data races at compile time, but logical race conditions still require careful design.

6.4 What is Grand Central Dispatch?

GCD is a low-level API that manages concurrent work via dispatch queues backed by a system-managed thread pool, so you submit blocks instead of creating threads. Key tools: DispatchQueue.main, DispatchQueue.global(qos:), async/sync, asyncAfter, DispatchGroup, DispatchSemaphore, DispatchWorkItem, and barriers. It remains relevant under the hood, but new code should prefer Swift Concurrency for safety.

6.5 What is the difference between serial and concurrent queues?

A serial queue runs one task at a time in FIFO order (useful to protect shared state); a concurrent queue may run many at once (still dequeued in FIFO order). The main queue is serial. Crucially, serial-vs-concurrent is orthogonal to async-vs-sync: async/sync is about whether the caller waits, not about how the queue runs work.

6.6 What are Quality of Service (QoS) classes?

QoS communicates priority and intent to the scheduler: .userInteractive (immediate, UI), .userInitiated (user is waiting), .utility (longer, with progress), .background (invisible, energy-efficient). Correct QoS improves responsiveness and battery. Avoid priority inversion by never blocking high-QoS work behind low-QoS work.

6.7 How do you cause and fix a deadlock with sync?

Calling queue.sync on the serial queue you are already running on deadlocks because it waits for itself; the classic case is DispatchQueue.main.sync from the main thread. Fix by using async, checking you are not already on the target queue, or restructuring to remove the synchronous cross-queue wait entirely.

6.8 What is the difference between GCD and OperationQueue?

OperationQueue is a higher-level API on top of GCD that gives Operation objects with dependencies, cancellation, maxConcurrentOperationCount, KVO-observable state, and priorities. Use Operations when you need dependency graphs, cancellation, or reusable encapsulated units of work; use raw GCD for lightweight one-off dispatch. For new code, async/await and task groups supersede both for most cases.

6.9 What is the difference between DispatchGroup and DispatchSemaphore?

A DispatchGroup tracks completion of many tasks and notifies when all finish (fan-out/fan-in). A DispatchSemaphore counts permits to limit access to a finite resource or bridge async to sync. Avoid using semaphores to block threads in modern async code; they defeat the cooperative thread pool and can cause priority inversion.

6.10 What is async/await and how does it improve on completion handlers?

async/await writes asynchronous code in sequential style; an async function suspends at await, freeing the thread, and resumes later. It removes the nested completion-handler "pyramid of doom," uses normal throws for errors, and eliminates forgotten-callback and double-callback bugs. Suspension does not block a thread; the cooperative pool reuses it, which also avoids thread explosion.

func loadProfile() async throws -> Profile {
    let (data, _) = try await URLSession.shared.data(from: url)
    return try JSONDecoder().decode(Profile.self, from: data)
}

Refs: WWDC21: Meet async/await in Swift, Swift Book: Concurrency

6.11 What is structured concurrency?

A model where the lifetime of concurrent child tasks is bound to a scope, so children cannot outlive their parent and cancellation/errors propagate predictably. async let and withTaskGroup create children awaited before the scope exits. This makes leaks and orphaned work structurally impossible and keeps reasoning local, in contrast to fire-and-forget threads.

Refs: WWDC21: Explore structured concurrency in Swift

6.12 What is the difference between async let and a TaskGroup?

async let runs a fixed, statically known set of concurrent children bound to locals, awaited later. A TaskGroup runs a dynamic number of children (one per item, added in a loop) and collects results. Both are structured; choose async let for a handful of parallel calls and a group for data-driven fan-out.

async let user = fetchUser()
async let posts = fetchPosts()
let screen = try await Screen(user: user, posts: posts)   // run in parallel

6.13 What is a Task, and the difference between structured and unstructured tasks?

Task { } starts unstructured work that is not tied to a scope; you manage its lifetime and cancellation through the returned handle. Task.detached additionally drops the surrounding actor context and priority inheritance. Prefer structured concurrency; use an unstructured Task mainly to bridge synchronous contexts (a button handler) into async, and avoid detached unless you truly need to escape the actor.

6.14 What is an actor?

An actor is a reference type that protects its mutable state by serializing access, so only one task runs its isolated code at a time, preventing data races without manual locks. External calls are awaited because they may suspend. Actors give "shared mutable state, safe by construction," replacing the error-prone pattern of guarding state with a private serial queue.

Refs: WWDC21: Protect mutable state with Swift actors

6.15 What is actor reentrancy and why is it a pitfall?

When an actor method awaits, it can suspend and let other tasks run on the actor, so state may change across the suspension point. You must re-check invariants after every await and not cache state across one. Reentrancy is what prevents actors from deadlocking, but it means "one task at a time" does not mean "one method runs to completion uninterrupted."

6.16 What is the MainActor?

A global actor that serializes work on the main thread. Annotate UI types or methods with @MainActor to guarantee main-thread execution, replacing manual DispatchQueue.main.async; await MainActor.run { } hops explicitly. SwiftUI view bodies and many UIKit APIs are implicitly main-actor-isolated, so the compiler now catches off-main UI mutation.

6.17 What is Sendable?

Sendable marks types safe to pass across concurrency domains without data races. Value types of Sendable members are automatically Sendable; classes must be immutable or internally synchronized and marked @unchecked Sendable with care; closures that cross boundaries are @Sendable. Swift 6 checks this strictly at compile time, turning "is this safe to share?" from a code-review question into a compiler guarantee.

Refs: WWDC22: Eliminate data races using Swift Concurrency, WWDC24: Migrate your app to Swift 6

6.18 How does cancellation work in Swift Concurrency?

Cancellation is cooperative: cancelling sets a flag, and the task must check Task.isCancelled or call try Task.checkCancellation() and stop. Many APIs (URLSession's async methods) throw CancellationError automatically, and structured children cancel when the parent does. It never forcibly kills running code, so long-running loops must poll for cancellation to be responsive.

6.19 How do you bridge callback-based APIs to async/await?

Use continuations: withCheckedContinuation/withCheckedThrowingContinuation wrap a callback so you call resume(returning:) or resume(throwing:) exactly once (resuming zero or twice is a bug the checked variants trap). For streams of values, use AsyncStream/AsyncThrowingStream. This is the standard adapter from legacy delegate/callback APIs into modern async code.

func value() async throws -> Data {
    try await withCheckedThrowingContinuation { cont in
        legacy { result in
            switch result { case .success(let d): cont.resume(returning: d)
                            case .failure(let e): cont.resume(throwing: e) }
        }
    }
}

6.20 What is AsyncSequence / AsyncStream?

AsyncSequence is the async analog of Sequence, iterated with for await. AsyncStream builds one from a closure that yields values over time (wrapping a delegate, notification, or callback source) with buffering and termination. It is the structured-concurrency way to model event streams, often replacing Combine for straightforward cases.

Refs: WWDC21: Meet AsyncSequence

6.21 How do you synchronize shared state without actors (legacy)?

Options: a serial dispatch queue as a lock substitute, NSLock/os_unfair_lock, DispatchSemaphore, or a reader-writer pattern (concurrent queue with .barrier writes). Actors are the preferred modern approach, but locks still matter for low-level or performance-critical code where actor hops are too costly.

6.22 What is a priority inversion?

A high-priority task waits on a resource held by a low-priority task that the scheduler is not running, stalling the important work. GCD mitigates this with priority donation, but you should avoid blocking high-QoS work on low-QoS work and avoid semaphores across QoS boundaries.

6.23 What is a run loop and where does it matter?

A run loop processes input sources and timers for a thread; the main run loop drives UI event handling and is where Timer and older source-based callbacks fire. Background threads need a run loop only for similar source-driven work. Understanding it explains why blocking the main thread freezes the UI and why timers need an active run loop.

7. SwiftUI

7.1 How does SwiftUI's rendering model work?

SwiftUI is declarative: you describe the UI as a function of state. When a source of truth changes, SwiftUI re-invokes the affected body, diffs the new view tree against the old using view identity, and applies the minimal updates to the underlying render tree. Views are lightweight value types recreated cheaply; the three core concepts are identity (which view is "the same" across updates), lifetime (how long state lives), and dependencies (what a view reads, which drives invalidation). Internalizing identity-lifetime-dependencies is the key to reasoning about SwiftUI correctness and performance.

Refs: WWDC21: Demystify SwiftUI, Apple: SwiftUI documentation

7.2 Is a View in SwiftUI a struct, and why value types?

Yes; View conformers are structs. SwiftUI keeps a cheap value description of the UI, diffs old versus new, and updates only what changed. Value semantics make recreation and comparison fast and predictable, and because state is stored separately (via property wrappers) rather than in the struct, recreating a view does not lose its state. This is the opposite of UIKit, where a view is a heavyweight reference object you mutate.

7.3 Explain the SwiftUI state property wrappers.

@State is local, view-owned mutable state for value types. @Binding is a two-way reference to state owned elsewhere. @StateObject creates and owns an ObservableObject once per view lifetime. @ObservedObject observes an object owned elsewhere (never create it inline, it can be recreated on redraw). @EnvironmentObject injects a shared object from the environment; @Environment reads environment values. With the Observation framework (iOS 17+), @Observable classes replace ObservableObject, used with @State and @Bindable. Picking the right wrapper is really a question of who owns the source of truth.

Refs: WWDC20: Data Essentials in SwiftUI

7.4 What is the difference between @StateObject and @ObservedObject?

@StateObject ties the object's lifetime to the view and instantiates it only once even if body re-runs. @ObservedObject does not own the object; if you create one inline with it, it is recreated whenever the parent redraws, silently losing state. Rule: the view that creates the object uses @StateObject; views that receive it use @ObservedObject (or @Bindable with Observation). This is one of the most common SwiftUI bugs in interviews and in production.

7.5 What is the Observation framework (iOS 17+)?

The @Observable macro makes a class's stored properties observable with fine-grained, per-property tracking. SwiftUI re-renders only views that actually read a changed property, which is more efficient than ObservableObject/@Published, where any published change invalidates all observers. It also removes @Published and objectWillChange boilerplate. The architectural win is precise dependency tracking, which directly reduces unnecessary body re-evaluations.

Refs: WWDC23: Discover Observation in SwiftUI

7.6 Why is the order of view modifiers important?

Each modifier wraps the current view and returns a new one, so order changes the result. .padding().background(.blue) paints the background around the padded area; .background(.blue).padding() paints behind the original content and then adds outside space. Thinking of modifiers as nested wrappers, not property setters, is the mental model that prevents layout surprises.

Text("Hi").padding().background(.blue)   // blue includes the padding
Text("Hi").background(.blue).padding()   // blue hugs text; padding is outside

7.7 When should you choose SwiftUI vs UIKit?

SwiftUI for new screens, rapid iteration, state-driven UI, and multiplatform reach; it handles dynamic type, dark mode, and accessibility with little code. UIKit when you need fine-grained control (complex collection layouts, custom interactive transitions), must support older OS versions, or need behavior SwiftUI does not yet expose. They interoperate, so the real-world answer is usually "SwiftUI-first, drop to UIKit where needed," not either/or.

7.8 How do you bridge UIKit and SwiftUI?

Embed SwiftUI in UIKit with UIHostingController. Embed UIKit in SwiftUI with UIViewRepresentable/UIViewControllerRepresentable, implementing makeUIView/updateUIView and using a Coordinator to route delegate and target-action callbacks back into SwiftUI state. The Coordinator is the crucial piece: it is how imperative UIKit callbacks become declarative SwiftUI state changes.

struct WebView: UIViewRepresentable {
    let url: URL
    func makeUIView(context: Context) -> WKWebView { WKWebView() }
    func updateUIView(_ view: WKWebView, context: Context) { view.load(URLRequest(url: url)) }
}

7.9 What is #Preview and how is it used?

The #Preview macro (or the older PreviewProvider) renders a view in Xcode's canvas without launching the app, across devices, color schemes, and seeded states, enabling fast visual iteration. Treat previews as a development tool that also documents the states a view supports.

7.10 How does SwiftUI decide when to re-render a view, and how do you keep it fast?

SwiftUI re-evaluates body when a dependency it reads changes, then diffs by identity and updates only changed nodes. Keep it fast by reading the minimum state a view needs (fine-grained @Observable), giving stable identities in lists (Identifiable), avoiding expensive work in body, and breaking large views into smaller ones so invalidation is localized. Performance problems are almost always over-broad dependencies causing too-wide invalidation.

Refs: WWDC23: Demystify SwiftUI performance

8. UIKit & App Lifecycle

8.1 Walk through the app launch sequence.

The system loads the executable and the runtime calls UIApplicationMain (or the SwiftUI @main App), which creates the UIApplication and the app delegate. application(_:didFinishLaunchingWithOptions:) does one-time setup; in scene-based apps the scene delegate then connects a UIWindowScene, creates the window, sets the root, and makes it key and visible, after which the app becomes active. Knowing this sequence tells you exactly where one-time setup, window construction, and per-scene UI belong.

8.2 What are the UIApplication states?

Not running, inactive (foreground but not receiving events, a brief transition), active (foreground, receiving events), background (executing off-screen, usually briefly), and suspended (in memory, not executing; purgeable under pressure). Transitions arrive via app/scene delegate callbacks; use them to save state, release resources, and pause work. SwiftUI surfaces this as @Environment(\.scenePhase).

8.3 What is the difference between the app delegate and the scene delegate?

Since iOS 13 the app delegate handles process-level events (launch, push registration, app-wide lifecycle) while the scene delegate handles UI lifecycle per window/scene (connect, foreground, background, disconnect), enabling multi-window iPad apps. SwiftUI abstracts both behind App, Scene, and scenePhase, but you still attach a delegate with @UIApplicationDelegateAdaptor for things like push registration.

8.4 Describe the UIViewController lifecycle.

init/loadView (build the view hierarchy), viewDidLoad (one-time setup after load), viewWillAppear (before each appearance), viewWillLayoutSubviews/viewDidLayoutSubviews (around layout), viewDidAppear, viewWillDisappear, viewDidDisappear. Put one-time wiring in viewDidLoad, refresh-on-show work in viewWillAppear, and never depend on frames before layout has run. The SwiftUI analog is onAppear/onDisappear/.task, but those can fire many times, so do not treat onAppear like viewDidLoad.

8.5 What is the responder chain?

A linked sequence of UIResponder objects (views, view controllers, the window, the application, the app delegate) through which events and actions propagate; if a responder does not handle an event it passes to next. It powers touch handling, nil-target actions that route to the first responder, and keyboard focus. becomeFirstResponder() inserts an object at the front to receive events (for example, to show the keyboard).

8.6 What is the difference between frame and bounds?

frame is the view's rectangle in its superview's coordinate system (position and size); bounds is the view's own coordinate system (origin usually (0,0)) and size. Changing bounds.origin scrolls content. Under a transform, frame becomes derived/undefined, so prefer bounds and center when rotating or scaling. This distinction is the root of many layout bugs once transforms are involved.

8.7 How does Auto Layout work, and what is intrinsic content size?

Auto Layout solves a system of linear constraints to compute frames. A view can expose an intrinsic content size (the size it prefers from its content, like a label's text), which participates with content-hugging (resists growing) and compression-resistance (resists shrinking) priorities; conflicts resolve by priority. Understanding priorities is what lets you build layouts that adapt gracefully instead of breaking or logging ambiguous-constraint warnings.

8.8 What causes scrolling jank and how do you fix it?

Main-thread work during scrolling: synchronous image decoding, expensive layout, offscreen rendering (shadows, masks, cornerRadius with masksToBounds on complex layers), and blocking I/O. Fixes: decode/downsample images off the main thread, reuse and pre-size cells, rasterize static layers, avoid blending and offscreen passes, and profile with Core Animation color-flags and the Hangs instrument. The principle is to keep the main run loop free during the 16ms (or less) frame budget.

Refs: WWDC21: Make blazing fast lists and collection views

8.9 What is cell reuse and why does it matter?

UITableView/UICollectionView recycle a small pool of cells via dequeueReusableCell, keeping memory bounded and avoiding constant allocation while scrolling. You must reset mutable state in prepareForReuse and configure fully on dequeue; otherwise stale content (especially from async image loads completing on the wrong row) appears. Reuse is a performance necessity that imposes a correctness obligation.

8.10 What is a diffable data source and compositional layout?

UICollectionViewDiffableDataSource/UITableViewDiffableDataSource apply NSDiffableDataSourceSnapshots of Hashable section/item identifiers, and UIKit animates the diff, removing manual performBatchUpdates and the index-mismatch crashes it caused. UICollectionViewCompositionalLayout builds complex layouts declaratively from items, groups, and sections. Together they bring a SwiftUI-like declarative feel to UIKit lists.

8.11 How do table views work, and data source vs delegate?

UITableViewDataSource supplies structure and cells (numberOfRowsInSection, cellForRowAt); UITableViewDelegate handles appearance and interaction (height, selection). Splitting them is interface segregation in practice: one protocol for "what to show," another for "how it behaves." For two tables on one controller, branch on the tableView parameter; for one table with two sources, switch on a mode or section.

8.12 How do you create custom views and handle gestures?

Subclass UIView, set up in the initializers, and override draw(_:) only for custom Core Graphics drawing (SwiftUI instead composes a View). Attach interaction with gesture recognizers (tap, pan, pinch, long-press, swipe) via target/action, ensuring isUserInteractionEnabled is true.

let tap = UITapGestureRecognizer(target: self, action: #selector(handleTap))
view.addGestureRecognizer(tap)

8.13 What are IBOutlet and IBAction, and the UIButton hierarchy?

@IBOutlet connects a storyboard/xib view to a code property; @IBAction connects a control event to a method. SwiftUI removes the need for both by describing the hierarchy in code bound to state. For the classic hierarchy question: UIButton -> UIControl -> UIView -> UIResponder -> NSObject.

8.14 Points vs pixels and asset scales.

Layout uses points, a device-independent unit; the screen scale maps points to physical pixels (2x or 3x on modern devices), and you ship @2x/@3x assets (or vector PDFs/SF Symbols). Designing in points with Auto Layout, size classes, and dynamic type is what makes one layout adapt across the device lineup.

9. Combine & Reactive

9.1 What is Combine?

Apple's declarative reactive framework built on Publishers (emit values over time), Subscribers (receive them), and Operators (transform streams), with demand-based backpressure through Subscription. Common types: Just, Future, PassthroughSubject, CurrentValueSubject, and @Published. It pairs naturally with SwiftUI and KVO/notification sources, though async/await and AsyncSequence increasingly replace it for one-shot work. Reach for Combine when you have continuous, composable streams (debounced search, combining several sources).

Refs: Apple: Combine documentation

9.2 What is the difference between PassthroughSubject and CurrentValueSubject?

Both are subjects you imperatively send into. CurrentValueSubject stores and replays its current value to new subscribers and exposes .value; PassthroughSubject stores nothing and only forwards events occurring after subscription. Use CurrentValueSubject to model state (there is always a current value) and PassthroughSubject to model events (things that just happen).

9.3 How do you avoid memory leaks in Combine?

Store subscriptions in a Set<AnyCancellable> owned by the object so they cancel on deallocation, and use [weak self] in sink/map closures referencing self. The cycle to watch: the subscription retains the closure, the closure retains self, and self retains the cancellable. Treating cancellables as owned lifetime, exactly like memory, is the discipline that keeps reactive code leak-free.

9.4 Combine vs async/await: how do you choose?

Use async/await for linear, one-shot asynchronous work (a network call, a sequence of steps) because it is simpler and integrates with structured concurrency and cancellation. Use Combine (or AsyncSequence) for streams of values over time and declarative composition (merging publishers, debouncing input, reacting to multiple sources). Many codebases use async/await as the default and Combine where genuine streams exist.

10. Networking & Web Services

10.1 How do you build a networking layer with URLSession?

Use URLSession with async/await (data(for:)), define a typed endpoint/request abstraction, decode JSON with Codable/JSONDecoder, map HTTP status codes to typed errors, and inject the session behind a protocol for testability. Layer in retry, caching policy, authentication with token refresh, and a background configuration for large transfers. The architectural goal is a thin, injectable, testable boundary so the rest of the app depends on domain models and typed errors, never on URLSession directly.

protocol HTTPClient { func send<T: Decodable>(_ r: Request<T>) async throws -> T }

func fetchUsers() async throws -> [User] {
    let (data, response) = try await URLSession.shared.data(from: usersURL)
    guard (response as? HTTPURLResponse)?.statusCode == 200 else { throw APIError.badStatus }
    return try JSONDecoder().decode([User].self, from: data)
}

Refs: WWDC21: Use async/await with URLSession, Apple: URLSession

10.2 What is Codable and how do you customize it?

Codable (Encodable & Decodable) enables automatic serialization to formats like JSON. Customize with CodingKeys for renaming, nested containers for restructuring, keyDecodingStrategy = .convertFromSnakeCase, dateDecodingStrategy for dates, and a custom init(from:)/encode(to:) for irregular payloads; for heterogeneous JSON, decode a discriminator field then branch. It replaces hand-written NSJSONSerialization parsing and NSCoding, moving serialization bugs to compile time.

struct Post: Codable {
    let id: Int; let title: String
    enum CodingKeys: String, CodingKey { case id; case title = "post_title" }
}

10.3 How do you wrap a completion-handler networking API in async/await?

The classic dataTask(with:completionHandler:) delivers (Data?, URLResponse?, Error?) on a background queue and forces manual optional handling. Wrap it in withCheckedThrowingContinuation, resuming exactly once, to expose a clean async API. New code should use the async methods directly and only wrap legacy or third-party callbacks.

10.4 Synchronous vs asynchronous requests, and why never block the main thread?

A synchronous request blocks its thread until completion; on the main thread that freezes the UI and risks a watchdog termination. Networking must always be asynchronous (async/await or callbacks) so the run loop stays free. This is non-negotiable: the main thread belongs to the user interface.

10.5 How do you handle networking errors and HTTP status codes?

Catch thrown URLErrors (offline, timeout), inspect HTTPURLResponse.statusCode to map 4xx/5xx to typed domain errors, and treat DecodingError separately so you can distinguish "server changed the contract" from "network failed." Present user-friendly messages, and retry only transient failures (with backoff). Good error taxonomy is what makes a networking layer debuggable in production.

10.6 How do you cancel an in-flight request?

With async/await, cancel the enclosing Task; URLSession's async calls are cancellation-aware and throw CancellationError/URLError(.cancelled). With the older API, call task.cancel(). Wiring cancellation to UI lifetime (for example cancelling a cell's image load on reuse, or a screen's .task on disappearance) prevents wasted work and wrong-row bugs.

10.7 How do you download and display an image efficiently?

In SwiftUI use AsyncImage. In UIKit, fetch off the main thread, downsample to the display size, cache in two tiers (NSCache in memory plus disk/URLCache), deduplicate concurrent requests for the same URL, and cancel on cell reuse. Decoding and downsampling are the expensive parts, so keeping them off the main thread is what preserves scroll performance.

AsyncImage(url: url) { $0.resizable().scaledToFill() } placeholder: { ProgressView() }

10.8 What URLSession configurations exist?

default uses the shared cache/cookies, ephemeral persists nothing to disk, and background continues uploads/downloads while the app is suspended (system-managed). Tasks include data, upload, download, and streaming. Choosing the right configuration is mostly about persistence and whether transfers must survive backgrounding.

10.9 What are the advantages of REST, and REST vs GraphQL?

REST is stateless, cacheable, and built on standard HTTP verbs and resource URLs, which is simple and scales well. GraphQL lets clients request exactly the fields they need from one endpoint, reducing over- and under-fetching at the cost of more server complexity and harder HTTP caching. This is the modern version of the old REST-vs-SOAP question (SOAP being heavier and XML-based); the choice is about client flexibility versus operational simplicity.

10.10 How do you handle authentication and token refresh?

Store tokens in the Keychain, attach the access token to requests, and on a 401 pause requests, refresh once (deduplicating concurrent refreshes behind a single shared task or an actor), then retry the failed requests. Never store tokens in UserDefaults, and use ASWebAuthenticationSession for OAuth flows. The deduplicated single-flight refresh is the detail interviewers look for, since naive code fires N refreshes for N concurrent 401s.

10.11 What is App Transport Security, and certificate pinning?

ATS enforces secure HTTPS (TLS 1.2+, forward secrecy) by default, with exceptions requiring justification in Info.plist. For high-security apps, pin the server's certificate or public key in urlSession(_:didReceive:completionHandler:) to defend against man-in-the-middle attacks, balancing security against the operational cost of rotating pins.

10.12 How do you integrate third-party libraries today?

Prefer Swift Package Manager, integrated into Xcode with semantic version rules and a Package.resolved lockfile; CocoaPods and Carthage are older alternatives common in legacy projects. SPM's native support, reproducible resolution, and module boundaries make it the default for new work.

10.13 What is URLSession.DataTaskPublisher?

Combine's dataTaskPublisher(for:) emits (data, response) and chains with map/decode/receive(on:), subscribed via sink. It is useful inside Combine pipelines, though async/await is now the simpler default for one-shot requests.

11. Notifications, Events & Background Execution

11.1 What is NotificationCenter and when do you use it?

A broadcast hub where objects post named notifications and any number of observers react; modern Swift uses the block-based observer (and can consume notifications as an async sequence). Use it for one-to-many, loosely coupled broadcast where the sender does not care who listens. The trade-off is traceability: broadcast is convenient but makes data flow harder to follow, so prefer explicit channels for core logic.

11.2 When do you use delegation vs closures vs notifications vs Combine?

Delegation: a defined contract with several related callbacks and a single owner. Closures: a single localized callback (completion handler), concise but watch capture cycles. NotificationCenter: broadcast to many unknown observers. Combine/async streams: continuous, composable value streams. Choose by cardinality (one vs many), coupling, and how many related events there are. Picking the lightest mechanism that fits is a hallmark of clean design.

11.3 What is the observer pattern in an iOS context, including KVO/KVC?

Objects subscribe to a subject and are notified on change. Implementations: NotificationCenter, KVO, Combine, and SwiftUI's state observation. KVC accesses properties by string key path; KVO observes changes on @objc dynamic properties via the block-based observe(_:options:) with type-safe Swift key paths. Modern Swift prefers Combine or the Observation framework over raw KVO except when interoperating with Objective-C frameworks.

class Account: NSObject { @objc dynamic var balance = 0.0 }
let token = account.observe(\.balance) { acc, _ in print(acc.balance) }

11.4 What is a push notification, and the end-to-end flow?

A way for a server to reach the app even when it is not running. Flow: the app registers and receives a device token, sends it to your server, and your server pushes a payload plus token to Apple Push Notification service (APNs), which delivers it to the device; you handle delivery and taps via UNUserNotificationCenterDelegate. In SwiftUI you register through a @UIApplicationDelegateAdaptor.

Refs: Apple: UserNotifications

11.5 What is the APNs payload size limit, and how do local notifications work?

Remote notification payloads are limited to 4 KB (larger for specific types like VoIP), much bigger than the old 256-byte limit. Local notifications are scheduled on-device via UNUserNotificationCenter with time, calendar, or location triggers, no server involved; rich content and background updates use notification service/content extensions and the content-available flag.

11.6 What are background modes and background tasks?

iOS strictly limits background execution. Options: BGAppRefreshTask (short, opportunistic refresh), BGProcessingTask (longer maintenance, often while charging), background URLSession (system-managed transfers), and specific modes (audio, location, VoIP). The system schedules these by usage patterns and energy, so you design for "eventually and opportunistically," never guaranteed timing.

12. Design Patterns

12.1 What is the delegate pattern?

A one-to-one pattern where an object hands off decisions or events to another conforming to a protocol; UIKit uses it heavily (UITableViewDelegate), and the delegate is held weak to avoid cycles. It fits when exactly one object should respond and there is a defined set of callbacks, contrasted with notifications (one-to-many) and closures (inline, anonymous).

protocol DownloadDelegate: AnyObject { func didFinish(_ data: Data) }
final class Downloader { weak var delegate: DownloadDelegate? }

12.2 What is the singleton pattern and what are its downsides?

A single globally shared instance (static let shared), appropriate for genuinely single resources (FileManager.default). Downsides: global mutable state, hidden dependencies, hard-to-test code, and concurrency hazards. The senior position is to prefer dependency injection and, if a singleton is justified, still inject it through a protocol so tests can substitute it. Singletons are a tool of last resort, not a default.

final class Session { static let shared = Session(); private init() {} }

12.3 What is dependency injection and why does it matter?

Supplying an object's dependencies from outside rather than constructing them internally, via initializer (preferred), property, or method injection. It decouples components, makes dependencies explicit, and lets tests substitute mocks/fakes. You rarely need a DI container; disciplined constructor injection plus a composition root is usually enough and keeps the graph visible.

12.4 What is the factory pattern?

Encapsulating object creation behind a method or type so callers depend on an abstraction, not concrete types. Useful for building view controllers, applying shared configuration, or returning different concrete types by input while keeping construction in one place. It pairs with DI as the "where objects get made" layer.

12.5 What are the Coordinator and Router patterns?

A coordinator owns navigation flow, removing routing from view controllers so they focus on their own UI and become reusable and testable; it decides the next screen and injects dependencies. In VIPER this is the Router. In SwiftUI, navigation state (a NavigationStack path) often plays the coordinating role. The core idea is that a screen should not know what comes after it.

12.6 What are the Adapter, Facade, and Decorator patterns?

Adapter wraps an incompatible interface to match what a client expects; Facade provides a simple interface over a complex subsystem (one service wrapping several APIs); Decorator wraps an object to add behavior without subclassing. All three favor composition over inheritance, which is why they appear constantly in well-factored Swift.

12.7 What is composition over inheritance?

Assemble behavior from small protocols and injected components rather than a deep class tree, which is more flexible, testable, and avoids fragile-base-class problems. In Swift, protocol extensions plus struct composition make this the default style rather than the exception.

protocol Engine { func start() }
struct Car { let engine: Engine }   // composes behavior instead of inheriting it

12.8 What is lazy loading?

Defer creating an expensive object until first use (lazy var, Core Data faulting, on-demand resources), saving work and memory when it is never needed. The trade-off is a one-time cost at first access and care around thread-safety, since lazy is not synchronized.

12.9 What is the Repository pattern, and how do you persist a model today?

A repository hides the data source (network, cache, database) behind a protocol so the app asks for domain models without knowing their origin, which eases testing and source swapping. To persist a simple model, encode it with Codable to JSON/Plist and write to disk, or store it in Core Data/SwiftData, rather than the old NSCoding/NSKeyedArchiver.

let data = try JSONEncoder().encode(user)
try data.write(to: fileURL, options: .atomic)

13. Architecture & App Structure

13.1 Explain MVC and Apple's variant.

MVC separates Model, View, and Controller. In Apple's UIKit the view controller often absorbs both view and controller roles, drifting into "Massive View Controller" as networking, formatting, and business logic accumulate. Mitigate by extracting view models, services, data sources, and child controllers. MVC is not wrong; the failure mode is letting the controller become a dumping ground.

13.2 Explain MVVM.

Model-View-ViewModel adds a ViewModel exposing presentation-ready state and view logic, with the View binding to it (via Combine, closures, or SwiftUI wrappers). The View stays thin and the ViewModel is unit-testable without UI, provided it does not import UIKit/SwiftUI. MVVM is the pragmatic default for most SwiftUI apps because SwiftUI's binding model fits it naturally.

13.3 Explain VIPER.

View, Interactor, Presenter, Entity, Router: View is passive UI, Interactor holds business logic and data access, Presenter formats data and handles actions, Entity is the model, Router handles navigation. It enforces strong separation and testability for large teams at the cost of significant boilerplate and many small types. It suits big, long-lived modules and is overkill for simple screens.

13.4 What is Clean / layered architecture?

Concentric layers with dependencies pointing inward: Entities (enterprise rules), Use Cases (application rules), Interface Adapters (presenters, gateways), and Frameworks/Drivers (UIKit, network, DB). Inner layers know nothing of outer ones; outer layers depend on abstractions defined inside. This isolates business logic from frameworks for testability and longevity, paying for it with indirection, so apply it where the domain is rich enough to justify the layers.

13.5 What is The Composable Architecture (TCA)?

A community framework (Point-Free) applying unidirectional data flow with State, Action, Reducer, and Effect, composed into features behind a single store. It emphasizes testability and predictable state, similar to Redux, at the cost of a learning curve and some overhead. It shines for complex, stateful features where deterministic, traceable state changes matter.

Refs: Point-Free: The Composable Architecture

13.6 How do you choose an architecture?

Match complexity to the problem: small apps thrive with MVC or MVVM; large multi-team apps benefit from clear module boundaries (Clean, VIPER, or feature modules). Optimize for testability, clear ownership of state, navigation separation, and onboarding cost, and avoid cargo-culting heavyweight patterns onto simple screens. Architecture should remove accidental complexity, not add it.

13.7 What is unidirectional data flow and why is it valuable?

State flows down, events flow up, and mutation happens in one place. This makes changes predictable and traceable (you can log every action), simplifies debugging, and aligns with SwiftUI. Two-way bindings can obscure where mutations originate, so constraining mutation to a reducer or store improves reasoning at scale.

13.8 What are SOLID principles in iOS practice?

Single Responsibility (split massive controllers), Open/Closed (extend via protocols and composition), Liskov Substitution (subtypes honor the supertype's contract), Interface Segregation (small focused protocols, like separate data source and delegate), and Dependency Inversion (depend on protocols, inject implementations). They are guidelines toward testable, maintainable design, not rigid laws; the spirit is loose coupling and clear responsibilities.

13.9 How do you modularize a large iOS app?

Split into Swift Packages or frameworks by feature and by layer (core, networking, design system, feature modules) with explicit public interfaces and dependency rules. Benefits: faster incremental builds, enforced boundaries, parallel team work, and reuse. Separating an interface package from its implementation breaks compile-time coupling and enables mocking, which is the key technique for keeping a large codebase fast to build and easy to test.

14. Data Persistence & Storage

14.1 What are the persistence options on iOS and when do you use each?

UserDefaults for small key-value preferences; Keychain for secrets; the file system (Documents, Application Support, Caches, tmp) for files and serialized data; Core Data or SwiftData for structured object graphs with relationships and queries; SQLite/GRDB for direct SQL control. Choose by data size, structure, query needs, security, and sync requirements. Matching the store to the data shape is the decision that keeps persistence simple instead of fighting the wrong tool.

14.2 What should you NOT store in UserDefaults?

Secrets and credentials (use the Keychain), large blobs or images (use files), and anything needing transactional integrity or complex queries (use a database). UserDefaults is unencrypted, loaded wholesale into memory, and not built for large or sensitive data, so misusing it leaks data and bloats memory.

14.3 What are the iOS app sandbox directories?

Documents/ for user-generated data that should be backed up; Library/Application Support/ for app data that is not user-facing (backed up); Library/Caches/ for regenerable data (not backed up, purgeable); tmp/ for short-lived files (system-purged). Mark large regenerable files with isExcludedFromBackup to respect iCloud limits. Choosing the right directory is also a backup and storage-pressure decision, not just organization.

14.4 What is Core Data, and is it a database?

Core Data is an object graph and persistence framework, not a database itself; it commonly uses SQLite as its store. It manages object life cycle, relationships, faulting (lazy loading), change tracking, validation, undo, and migrations, and you work with managed objects rather than SQL. Thinking of it as "an object graph manager that happens to persist" explains both its power and its threading rules.

14.5 Explain the Core Data stack.

NSManagedObjectModel (the schema), NSPersistentStoreCoordinator (mediates contexts and stores), NSManagedObjectContext (an in-memory scratchpad for objects), and the persistent store (SQLite). NSPersistentContainer bundles this. Objects live in a context, and saving pushes changes through the coordinator to the store. Understanding the context as a scratchpad is what makes the concurrency rules make sense.

14.6 How do you use Core Data correctly across threads?

A context is bound to a queue; never share managed objects across contexts/threads. Use the viewContext on the main queue for UI and performBackgroundTask/a private-queue context for heavy work, touching contexts only inside perform/performAndWait. Pass NSManagedObjectIDs between contexts, not objects, and merge changes (notifications or automaticallyMergesChangesFromParent). Violating the queue confinement rule is the most common Core Data crash.

container.performBackgroundTask { ctx in
    let book = Book(context: ctx); book.title = "New"
    try? ctx.save()   // background import, then merge into viewContext
}

14.7 What is a fault in Core Data?

A fault is a placeholder for an object or relationship whose data has not been loaded; Core Data fires it (loads data) on first access to conserve memory. Excessive faulting in loops causes performance problems, mitigated with fetchBatchSize, relationship prefetching, and batch operations. Faulting is the lazy-loading mechanism that makes large graphs memory-efficient until you accidentally trigger it N times in a loop.

14.8 How do Core Data migrations work?

Lightweight migration handles simple schema changes automatically by inferring a mapping when you version the model; complex changes need a mapping model, custom policies, or staged progressive migrations through intermediate versions. Always version the model and test migrations against real production data shapes, because a failed migration corrupts or loses user data.

14.9 How do you improve Core Data performance?

Use background contexts for writes; batch operations (NSBatchInsertRequest, NSBatchUpdateRequest, NSBatchDeleteRequest) that bypass loading objects; set fetchBatchSize/fetchLimit; prefetch relationships; index frequently queried attributes; and drive table/collection views with NSFetchedResultsController for efficient, change-based updates. The theme is "do not load what you do not need, and do not touch the main thread for heavy work."

14.10 What is SwiftData?

SwiftData (iOS 17+) is Apple's modern persistence framework layered on Core Data, using Swift macros. Declare models with @Model, configure a ModelContainer, work through a ModelContext, and query with @Query in SwiftUI or FetchDescriptor with #Predicate elsewhere. It removes the .xcdatamodeld editor and much boilerplate while keeping Core Data's capabilities, and its type-safe API catches schema mistakes at compile time.

@Model final class Book { var title: String; init(title: String) { self.title = title } }
struct BookList: View { @Query var books: [Book]; var body: some View { List(books) { Text($0.title) } } }

Refs: WWDC23: Meet SwiftData, WWDC23: Build an app with SwiftData

14.11 How does SwiftData compare to Core Data?

SwiftData offers a cleaner, type-safe, Swift-native API with tight SwiftUI integration and less boilerplate; Core Data is more mature, exposes lower-level control (custom migrations, fine-grained fetch tuning, advanced concurrency), and supports older OS versions. They share a storage foundation and can interoperate. Choose SwiftData for new iOS 17+ apps unless you need Core Data's advanced control.

14.12 How does concurrency work in SwiftData?

ModelContext is not Sendable and is tied to its actor/context; the main context backs UI. For background work use a context on a background actor (the @ModelActor macro builds an actor that owns its own context), and pass PersistentIdentifiers across boundaries, never model instances. The rules mirror Core Data's queue confinement, now expressed through actors.

14.13 How do you handle data caching?

Layer caches: in-memory (NSCache, which is thread-safe and evicts under pressure) for speed, disk cache for persistence across launches, and HTTP caching via URLCache and proper cache-control headers. Define eviction (cost/count limits) and freshness/invalidation (TTL, ETags), and always treat caches as discardable. The discipline is that a cache must never be the source of truth.

14.14 How do you keep data in sync with a server?

Strategies: last-write-wins, version vectors/timestamps for conflict detection, delta sync via a sync token, and offline queues for outgoing changes replayed on reconnect. Persist a local source of truth, reconcile on reconnect, and surface conflicts deliberately. NSPersistentCloudKitContainer provides built-in sync within the Apple ecosystem. Sync is fundamentally a conflict-resolution problem, so decide the policy before the transport.

14.15 How do you securely store sensitive data at rest?

Use the Keychain for secrets, rely on Data Protection (files encrypted while the device is locked, configurable via protection classes), never log sensitive data, and consider encrypting databases (SQLCipher) for sensitive structured data, tying access to device unlock or biometrics where appropriate. Security at rest is a layered default, not a single API call.

Refs: Apple: Keychain Services

14.16 What is NSCache and how does it differ from a Dictionary?

NSCache is a mutable, thread-safe, key-value cache that automatically evicts under memory pressure and supports cost/count limits; unlike a Dictionary it is purgeable and concurrency-safe by design, ideal for image and computed-result caches. It does not persist across launches, so it is a memory tier, not storage.

14.17 What are @AppStorage and @SceneStorage, and NSManagedObjectID?

@AppStorage reads/writes UserDefaults and refreshes the view on change; @SceneStorage persists small per-scene UI state for restoration; both replace manual UserDefaults access for simple values. NSManagedObjectID uniquely identifies a managed object; a temporary ID becomes permanent after a save, and the permanent ID's URI can be stored and resolved later, which is how you safely reference objects across contexts and launches.

15. Testing

15.1 What kinds of tests should an iOS app have?

A testing pyramid: many fast unit tests (pure logic, view models, services), fewer integration tests (multiple components, persistence, stubbed networking), and a small number of UI/end-to-end tests (XCUITest) for critical flows. The pyramid balances coverage and confidence against speed and flakiness; UI tests are slow and brittle, so keep them few and high-value. Test distribution is an architecture decision as much as a QA one.

15.2 What is the difference between XCTest and Swift Testing?

XCTest uses XCTestCase subclasses, the XCTAssert family, and setUp/tearDown. Swift Testing (2024) uses the @Test macro, #expect/#require with expressive failure output, @Suite types, parameterized tests, traits, and native async/await. New projects increasingly adopt Swift Testing, and both can coexist during migration.

import Testing
@Test func addsNumbers() { #expect(2 + 3 == 5) }

Refs: WWDC24: Meet Swift Testing

15.3 What makes a good unit test?

Fast, isolated (no network/disk/global state), deterministic (no reliance on time, ordering, or randomness), focused on one behavior, and readable (Arrange-Act-Assert). It should fail for exactly one reason and have a name describing scenario and expected outcome, and it should test observable behavior, not implementation details. Tests that assert on internals break on every refactor and erode trust.

15.4 How do you make code testable?

Depend on protocols and inject dependencies (clock, network, persistence) so you can substitute doubles; keep business logic out of view controllers and free of UIKit imports; avoid singletons and static state; make side effects explicit; and isolate nondeterminism (dates, UUIDs, randomness) behind injectable providers. Testability and good architecture are the same property viewed from two angles.

15.5 What are the types of test doubles?

Dummy (passed but unused), stub (returns canned data), spy (records how it was called), mock (verifies expected interactions), and fake (a working lightweight implementation, like an in-memory store). Swift has no built-in mocking runtime, so you hand-write protocol conformances, which keeps doubles explicit and type-safe.

15.6 How do you test networking code?

Inject a session abstraction or subclass URLProtocol to intercept and stub requests, returning fixed responses and errors without hitting the network. Test decoding, error mapping, retry, and status-code handling; keep tests offline and deterministic, reserving real calls for separate, tagged integration tests. Stubbing at the URLProtocol boundary lets you test the real URLSession code path while staying hermetic.

15.7 How do you test asynchronous, concurrent, and actor code?

For async, make the test method async and await directly (or use XCTestExpectation for callbacks); avoid arbitrary sleep and await deterministic signals. For actors and concurrency, await actor calls, inject clocks to remove wall-clock waits, drive concurrency with task groups, and run with Thread Sanitizer plus the Swift 6 checker to surface data races. Determinism is the whole game in concurrent tests.

15.8 What is snapshot testing, and how do you read code coverage?

Snapshot testing records a reference rendering (image or serialized text) and fails on divergence, effective for catching unintended UI changes across many states but sensitive to device/OS and requiring intentional reference updates. Code coverage (enabled in the scheme) shows which lines/branches ran; treat it as a guide to find untested areas, not a target, since high coverage of trivial code gives false confidence.

15.9 How do you reduce test flakiness, and what are XCUITest's limits?

Remove shared mutable state, control time and randomness via injection, avoid real network/timers, await conditions instead of sleeping, and reset persistence between tests. XCUITest drives the app through the accessibility layer to tap and assert on real flows but is slow, out-of-process, and flaky; use stable accessibility identifiers, launch arguments to stub backends and seed state, and reserve it for a few critical journeys.

16. Xcode, Build & Tooling

16.1 What is the difference between a target, a scheme, and a build configuration?

A target produces one product (app, framework, test bundle) with its own settings and files. A scheme defines how targets are built, run, tested, profiled, and archived, and which configuration each action uses. A build configuration (Debug, Release, custom) is a named set of build settings selected per action. Together they control what gets built and how, and confusing them is a common source of "works on my machine" issues.

16.2 What is the difference between Debug and Release builds?

Debug disables optimization (-Onone), includes full symbols and assertions, and favors fast iteration. Release enables optimization (-O/whole-module), strips or externalizes symbols, and drops debug-only checks for speed and size. Bugs that appear only in Release are usually optimization- or assertion-related (for example, relying on a side effect inside an assert, which is stripped in Release).

16.3 How do you manage dependencies in iOS?

Swift Package Manager (Apple's integrated, preferred solution), CocoaPods (mature, centralized, edits the workspace), and Carthage (decentralized, prebuilt frameworks). SPM is the default for new projects given native Xcode support, semantic version ranges, and reproducible resolution via Package.resolved. Fewer, well-audited dependencies is itself an architectural choice that affects build time and risk.

16.4 What is an .xcconfig file, and what are Info.plist and entitlements?

An .xcconfig defines build settings in plain text outside the Xcode UI, enabling per-environment config, clean diffs, and shared settings. Info.plist declares app metadata and configuration (bundle ID, version, permission usage descriptions, orientations, ATS exceptions). Entitlements grant capabilities requiring system authorization (push, App Groups, Keychain sharing, iCloud, Sign in with Apple), tied to the provisioning profile. These files are where environment and capability configuration lives, so they belong in version control and review.

16.5 What is the difference between print, os_log, and Logger?

print is unstructured and debug-only. The unified logging system (os_log and the Swift Logger) is structured, categorized by subsystem/category, performant, privacy-aware (dynamic data redacted by default), and viewable in Console.app and Instruments even in production. Prefer Logger for real apps, and be deliberate about privacy annotations so you do not leak PII into logs.

16.6 How do you symbolicate a crash report?

Match the crash report to the build's dSYM so addresses map to function names and line numbers; Xcode Organizer symbolicates App Store/TestFlight crashes automatically, otherwise use symbolicatecrash/atos with the dSYM. Archive and retain dSYMs for every release, because without them production crash reports are unreadable.

16.7 What are sanitizers in Xcode?

Runtime diagnostics: Thread Sanitizer (data races), Address Sanitizer (use-after-free, overflows), and Undefined Behavior Sanitizer, enabled in the scheme's Diagnostics. They add overhead but catch hard-to-find bugs; running TSan in CI for concurrent code is one of the highest-leverage habits for stability.

16.8 What does a CI/CD and release pipeline look like?

CI (Xcode Cloud, GitHub Actions, Bitrise, GitLab, Jenkins) builds and tests every change; Fastlane automates building, signing, screenshots, and uploads. Release flow: bump version/build and signing, archive (Product > Archive), validate and upload to App Store Connect, fill metadata and screenshots, submit for review, and distribute betas via TestFlight before public release. Automating signing and upload is what makes releases boring, which is the goal.

17. Performance & Debugging

17.1 What is the general approach to performance optimization?

Measure first with Instruments to find the real bottleneck, optimize the hottest path, then re-measure to confirm, avoiding premature optimization. Set budgets (launch time, frame rate, memory), test on the lowest supported device, and treat performance as a measured, regression-tested requirement. "Measure, fix the top item, measure again" beats guessing every time.

17.2 What common Instruments do you use and what do they measure?

Time Profiler (CPU by stack), Allocations and Leaks (memory growth and cycles), the Memory Graph Debugger (retain cycles), Core Animation (rendering and frame rate), Energy, Network, and the Hangs/SwiftUI instruments. Instruments exist so you optimize against evidence, not intuition.

17.3 How do you reduce app launch time?

Minimize work in didFinishLaunching and initial view setup, defer non-critical initialization, reduce dynamic library count and static initializers, lazy-load heavy resources, and profile with the App Launch instrument. Distinguish cold from warm launch and stay well under the watchdog limit; pre-main time dominated by too many dylibs is a frequent culprit.

17.4 How do you find and fix UI hangs?

A hang is the main thread blocked beyond a perceptible threshold. Find them with the Hangs and Time Profiler instruments and Xcode's hang detection, then move synchronous I/O, heavy parsing, image decoding, and expensive layout off the main thread, coalesce work, and keep the run loop free. If the app talks to a server, also check for a synchronous or never-returning network call with a proxy.

17.5 How do you reduce app binary size and optimize memory?

For size: enable app thinning and on-demand resources, strip unused symbols and architectures, right-size and compress assets (HEIC, asset catalogs), remove dead code, and audit large dependencies. For memory: downsample images to display size, cap caches with NSCache, release caches on warnings, stream large files instead of loading wholesale, and eliminate retain cycles; jetsam terminates apps that exceed limits, so peak memory is a hard constraint.

17.6 How do you debug a memory leak?

Use the Memory Graph Debugger to find objects that should be gone but persist, the Leaks instrument to detect leaked allocations, and Allocations to track growth, then look for strong cycles in closures, delegates, and Combine subscriptions and confirm with deinit logging. Reproducing by repeatedly entering and leaving a screen while watching the live count is the fastest way to surface a controller that never deallocates.

17.7 LLDB pointer scenario: how do you reason about it?

Given variables and pointers in scope at a breakpoint, predict debugger output: po x prints a value, po &x prints its address, a pointer variable prints the address it holds, po &ptr prints the pointer's own address, and po *ptr dereferences to the pointed-to value; p, frame variable, and expression behave similarly. The exercise tests whether you understand the value/address/dereference distinction, which underlies all pointer debugging.

17.8 How do you debug a low-memory crash after repeated present/dismiss, and EXC_BAD_ACCESS?

For the low-memory crash, suspect a retain cycle or leaked controller: take Allocations heap snapshots before and after a present/dismiss cycle and inspect the delta, or use the Memory Graph Debugger to find retained controllers, then break the offending strong reference. For EXC_BAD_ACCESS, enable Address Sanitizer and Zombie Objects to catch use-after-free, and audit unowned references and unsafe pointer use. Both reduce to "find who still holds, or who already freed, this memory."

17.9 How do you use breakpoints and the view hierarchy debugger effectively?

Use symbolic and conditional breakpoints to stop only when relevant, action breakpoints to log without editing code, and watchpoints for value changes; use the Debug View Hierarchy to inspect the live 3D view tree for clipping, zero-size frames, and z-ordering issues. Knowing these turns "add print statements and rerun" into targeted, single-pass investigation.

18. System Design Scenarios

18.1 How would you design an image-loading and caching library?

Expose an async API keyed by URL; check a two-tier cache (in-memory NSCache, then disk) before fetching; deduplicate concurrent requests for the same URL behind a single shared task; decode and downsample off the main thread; cancel a request when its cell is reused; respect HTTP caching headers; and bound memory with cost limits and eviction. Make the network layer injectable for testing. The defining details are request deduplication, downsampling, and cancellation, which separate a real implementation from a naive one.

18.2 How would you design an offline-first feed?

Persist a local source of truth (Core Data/SwiftData), render from it immediately on launch, and sync in the background using a sync token for deltas. Queue user actions while offline and replay on reconnect with explicit conflict resolution, and drive the UI from change notifications (NSFetchedResultsController or @Query) with pagination and clear sync/error states. Offline-first is really "the local store is the truth and the network is an updater," which inverts the typical request-render model.

18.3 How would you design an analytics or logging pipeline in an app?

Define a typed event protocol, buffer events locally, batch-upload on a schedule or on backgrounding, and retry with backoff, persisting the queue so events survive termination. Abstract the transport behind a protocol so you can add providers and test without network, and respect privacy and consent while avoiding PII. Durability (events survive crashes) and a clean provider abstraction are the parts interviewers probe.

18.4 How do you approach a flaky, hard-to-reproduce production crash?

Gather data first: symbolicated crash reports, breadcrumbs/logs, and segmentation by OS/device/version. Form hypotheses from the stack and recent changes, add targeted non-intrusive logging or feature flags, and try to reproduce under sanitizers. Fix the root cause, add a regression test, and verify the crash rate drops in the next release. Resist "fix by guessing"; let data narrow the search space.

18.5 How do you keep an app maintainable as the team and codebase grow?

Establish clear module boundaries and ownership, enforce architecture and style via linters and code review, invest in a design system and shared infrastructure, keep a healthy test suite and CI, document decisions (ADRs), and pay down tech debt continuously. Optimize for readability and onboarding over cleverness. Maintainability is mostly social and structural, not a single technical trick.

18.6 How do you stay current with the iOS platform?

Watch WWDC sessions, read release notes and Swift Evolution proposals, follow the Swift forums and reputable community resources, experiment in sample projects, and adopt new APIs deliberately where they cut complexity or improve performance, balanced against minimum-deployment-target constraints. Adopting new tech for its own sake is a smell; adopting it to delete code or fix a real problem is the right instinct.

19. Data Structures & Algorithms

19.1 How are Swift Array, Dictionary, and Set implemented, and their performance?

Array is a contiguous, ordered, copy-on-write buffer with O(1) indexed access and amortized O(1) append; Dictionary and Set are hash tables with average O(1) lookup/insert/delete. All are value types with COW, so copies are cheap until mutated. Knowing the underlying representation explains both the complexity guarantees and why value semantics do not cost a full copy on every pass.

19.2 When do you use Array vs Set vs Dictionary?

Array for ordered sequences and indexed access, Set for fast membership and uniqueness, and Dictionary for key-to-value lookup. Choosing Set/Dictionary over scanning an array turns O(n) membership checks into O(1), which is the most common practical algorithmic win in app code.

19.3 What makes a good Hashable conformance?

A good hash distributes values evenly and stays consistent with == (equal values must hash equally), and should not include mutable fields that change after the value is inserted into a set/dictionary. Violating the equal-implies-equal-hash rule corrupts hashed collections in ways that are painful to debug.

19.4 Swap two variable values without a third variable.

Idiomatic Swift uses a tuple; the arithmetic trick is the classic interview answer.

var x = 10, y = 5
(x, y) = (y, x)             // idiomatic
// arithmetic: x = x + y; y = x - y; x = x - y

19.5 Reverse a linked list.

Walk the list, repointing each node's next to the previous node, which is O(n) time and O(1) space.

final class Node { var value: Int; var next: Node?; init(_ v: Int) { value = v } }
func reverse(_ head: Node?) -> Node? {
    var prev: Node? = nil, curr = head
    while let node = curr { curr = node.next; node.next = prev; prev = node }
    return prev
}

19.6 Find the second-minimum element in an array (O(n)).

Track the smallest and second-smallest in a single pass.

func secondMin(_ a: [Int]) -> Int? {
    var first = Int.max, second = Int.max
    for x in a {
        if x < first { second = first; first = x }
        else if x < second && x != first { second = x }
    }
    return second == Int.max ? nil : second
}

19.7 Ball/color exercise: contains and filter.

Use idiomatic collection APIs rather than manual loops.

enum BallColor { case red, black, white }
struct Ball { let color: BallColor }
func containsRedBall(_ balls: [Ball]) -> Bool { balls.contains { $0.color == .red } }
func redBalls(_ balls: [Ball]) -> [Ball] { balls.filter { $0.color == .red } }

19.8 Shuffle a deck and pick a random element.

Swift provides both; shuffle() is Fisher-Yates under the hood, which is the correct unbiased shuffle (naive swap-each-with-random is biased).

var deck = Array(1...52)
deck.shuffle()
let card = deck.randomElement()

19.9 Free-shipping problem: two prices closest to a threshold.

Find two distinct prices whose sum is as close as possible to (without exceeding) the threshold. Sorting plus two pointers is O(n log n); a hash set of complements gives O(n).

func twoForFreeShipping(_ prices: [Int], threshold: Int) -> (Int, Int)? {
    let sorted = prices.sorted()
    var lo = 0, hi = sorted.count - 1, best: (Int, Int)? = nil, bestSum = Int.min
    while lo < hi {
        let sum = sorted[lo] + sorted[hi]
        if sum <= threshold { if sum > bestSum { bestSum = sum; best = (sorted[lo], sorted[hi]) }; lo += 1 }
        else { hi -= 1 }
    }
    return best
}

20.1 Quick-reference comparisons

A fast recall sheet for the most common "compare X and Y" prompts:

  • Struct vs Class: value vs reference; copy vs shared; no inheritance vs inheritance; identity only on classes; default to struct.
  • weak vs unowned: optional and auto-nil (safe) vs non-optional, assumes lifetime, crashes if wrong.
  • some vs any: opaque single concrete type with static dispatch vs existential box with dynamic dispatch.
  • @StateObject vs @ObservedObject: creates and owns (once per lifetime) vs observes an externally owned object.
  • ObservableObject vs @Observable: publish-and-invalidate-all vs fine-grained per-property tracking (iOS 17+).
  • GCD vs async/await: manual queues and callbacks vs structured, suspending, cancellation- and race-aware.
  • Actor vs serial queue: language-level isolation with reentrancy and Sendable checking vs manual queue discipline.
  • Core Data vs SwiftData: mature and lower-level with broad OS support vs modern, macro-based, SwiftUI-native (iOS 17+).
  • MVC vs MVVM vs VIPER: simple but risks massive controllers vs testable view models vs maximal separation with boilerplate.
  • NotificationCenter vs delegate vs closure vs Combine: one-to-many broadcast vs one-to-one contract vs single inline callback vs composable streams.
  • REST vs GraphQL: standard verbs and easy caching vs exact-field queries with more server complexity.
